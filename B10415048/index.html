Locking

1.前言

由於xv6是跑在多CPU上的作業系統，這些CPU都使用相同的RAM並分享其中的數據，所以xv6必須建立一種機制來防止他們之間互相干擾，即便只有一個CPU也需要防止中斷處理程序跟非中斷之間相互影響，xv6為了這兩種情況，使用了鎖(locking)來保證一個時間點只會有一個CPU持有鎖，如果只能在持有特定鎖時才能使用裡面的數據，就能保證同一時間只有一個CPU會使用裡面的數據，這樣就可以用鎖來保護裡面的數據結構。

2.Race conditions

下面這個例子會講解，為何要用到鎖。
假設有一段程式碼如下圖

1 struct list {
2 int data;
3 struct list *next;
4 };
5
6 struct list *list = 0;
7
8 void
9 insert(int data)
10 {
11 struct list *l;
12
13 l = malloc(sizeof *l);
14 l->data = data;
15 l->next = list;
16 list = l;
17 }

看起來沒有甚麼問題，但是如果一旦有兩個不同CPU同時來執行insert就會發生問題，假設兩者都執行到15行，而都還未執行第16行。這樣的話就會出現兩個list node，並且next都會被設置成list，當兩者都執行第16行後，後面執行的會蓋掉前面執行的，於是前面執行新增的node就會遺失，這種問題就被稱為Race conditions。
所以這個時候我們就可以使用鎖來防止這種情況，因為一段時間只會有一個CPU來執行程式，就可以避免上述的狀況。

3.資料結構

struct spinlock {
    uint locked; // 鎖如果是可使用的話是0，非0則是正在被其他CPU用
    //下面是為了方便debug
    char *name; // 鎖的名字
    struct cpu *cpu; // 看是哪個CPU持有鎖
    uint pcs[10]; // program counter的陣列
};

spinlock.c

void acquire(struct spinlock*);
void getcallerpcs(void*, uint*);
int holding(struct spinlock*);
void initlock(struct spinlock*, char*);
void release(struct spinlock*);
void pushcli(void);
void popcli(void);

4.實作lock

void initlock(struct spinlock *lk, char *name)
{
    //初始化lock
    lk−>name = name;
    lk−>locked = 0;
    lk−>cpu = 0;
}

為了讓他實現鎖的功能，xv6依賴於一個特殊的x86指令xchg，xchg交換了記憶體中的字組(word)跟暫存器(register)的內容(contents)，function acquire 會反覆的使用xchg，每一次都讀取 lk->locked 然後設置為1。如果鎖已經被持有了，lk->locked 就已經為1了，故 xchg 會返回1然後繼續循環。如果 xchg 返回0，但是 acquire 已經成功獲得了鎖，即 locked 已經從0變為了1，這時循環可以停止了。一旦鎖被獲得了，acquire會記錄獲得鎖的CPU和stack trace。

xchg(volatile uint *addr, uint newval)
{
    uint result;
    asm volatile("lock; xchgl %0, %1" :
    "+m" (*addr), "=a" (result) :
    "1" (newval) :
    "cc");
    return result;
}

void acquire(struct spinlock *lk)
{
    pushcli(); // 停用中斷來避免鎖死
    if(holding(lk))
        panic("acquire");
    while(xchg(&lk−>locked, 1) != 0); //一直循環，直到獲得鎖。
    __sync_synchronize();
     //記錄獲得鎖的 CPU和stack trace
    lk−>cpu = mycpu();
    getcallerpcs(&lk, lk−>pcs);
}

release則做跟acquire相反的事情：清除 debugging fields並且把鎖給放開。

void release(struct spinlock *lk)
{
    if(!holding(lk))
        panic("release");
    lk−>pcs[0] = 0;
    lk−>cpu = 0;
    __sync_synchronize();  
    asm volatile("movl $0, %0" : "+m" (lk−>locked) : );   
    popcli();
}

5.Interrupt handlers

Xv6在許多情況下整個系統會鎖死，為了避免這種情況，當中斷處理程序會使用某個鎖時，處理器就不能在允許中斷發生時持有鎖.xv6做得更絕：允許中斷時不能持有任何鎖它使用pushcli和popcli來屏蔽中斷（cli是x86屏蔽中斷的指令）.acquire在嘗試獲得鎖之前調用了pushcli，釋放則在釋放鎖後調用了popcli.pushcli和 popcli不僅包裝了cli和sti，它們還做了計數工作，這樣就需要調用兩次popcli來抵消兩次pushcli;這樣，如果代碼中獲得了兩個鎖，那麼只有當兩個鎖都被釋放後中斷才會被允許。

獲得一定要在可能獲得鎖的xchg之前調用pushcli。如果兩個顛倒了，就可以在幾個時鐘週期裡，中斷仍被許可，而鎖也被獲得了，如果這時不幸地發生了 中斷，系統就會死鎖。類似的，發布也一定要在釋放鎖的xchg之後調用popcli。

另外，中斷處理程序和非中斷代碼對彼此的影響也讓我們看到了遞歸鎖的缺陷。如果xv6使用了遞歸鎖（即如果CPU獲得了某個鎖，那麼同一CPU上可以再次獲得該鎖）那麼中斷處理程序就可能在非中斷代碼正運行到臨界區時運行，這樣就非常混亂了。當中斷處理程序運行時，它所依賴的不變量可能暫時被破壞了。例如ideintr會假設未處理請求鍊錶是完全的。若xv6使用了遞歸鎖，ideintr就可以在iderw正在修改鍊錶，這樣ideintr就會使用這個不正確的鍊錶。

void pushcli(void)
{
    int eflags;
    eflags = readeflags();
    cli();
    if(mycpu()−>ncli == 0)
    mycpu()−>intena = eflags & FL_IF;
    mycpu()−>ncli += 1;
}

void popcli(void)
{
    if(readeflags()&FL_IF)
    panic("popcli − interruptible");
    if(−−mycpu()−>ncli < 0)
    panic("popcli");
    if(mycpu()−>ncli == 0 && mycpu()−>intena)
    sti();
}
