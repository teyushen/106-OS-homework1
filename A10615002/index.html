Page tables

1.運作原理
操作系統通過頁表機制實現了對內存空間的控制。x86的指令計算的都是虛擬地址，機器的RAM或者物理內存，則是用物理地址來作標記的，x86 的頁表硬件通過映射機制將虛擬地址和物理地址聯系起來。
一個頁表在物理內存中像一棵兩層的樹。樹的根是一個4096字節的頁目錄，其中包含了1024個類似 PTE 的條目，但其實每個條目是指向一個頁表頁的引用。而每個頁表頁又是包含1024個32位PTE的數組。分頁硬件使用虛擬地址的高10位來決定對應頁目錄條目。如果想要的條目已經放在了頁目錄中，分頁硬件就會繼續使用接下來的10位來從頁表頁中選擇出對應的PTE。否則，分頁硬件就會抛出錯誤。每個PTE都包含一些標志位，說明分頁硬件對應的虛擬地址的使用權限，xv6設置了PTE中的PTE_U 、PTE_W、PTE_P標志位。
每個進程都有自己的頁表，xv6會在進程切換時通知分頁硬件切換頁表。進程的用戶內存從0開始，最多能夠增長到KERNBASE, 這使得一個進程最多只能使用2GB的內存。xv6在每個進程的頁表中都包含了內核運行所需要的所有映射，而這些映射都出現在KERNBASE之上。
每個進程的頁表同時包括用戶內存和內核內存的映射，這樣當用戶通過中斷或者系統調用轉入內核時就不需要進行頁表的轉換了。大多數情況下，內核都沒有自己的頁表，所以內核幾乎都是在借用用戶進程的頁表。xv6保證了每個進程只能使用其自己的內存，並且每個進程所看到的內存都是從虛擬地址0開始的一段連續內存。對于一個進程，xv6只把該進程所使用的內存對應的 PTE 的 PTE_U 設為 1，其他 PTE 則不然，這樣就可以實現前者。對于後者，則是讓頁表把連續的虛擬頁映射到實際分配的物理頁。

3.代碼解釋
vm.c

typedef uint pde_t;

pde_t *kpgdir;

static pte_t *walkpgdir(pde_t *pgdir, const void *va, int alloc) {
pde_t *pde;
pte_t *pgtab;

pde = &pgdir[PDX(va)];
if(*pde & PTE_P) {
    pgtab = (pte_t*)P2V(PTE_ADDR(*pde));
} else {
    if(!alloc || (pgtab = (pte_t*)kalloc()) == 0)
        return 0;
    // Make sure all those PTE_P bits are zero.
    memset(pgtab, 0, PGSIZE);
    // The permissions here are overly generous, but they can
    // be further restricted by the permissions in the page table
    // entries, if necessary.
    *pde = V2P(pgtab) | PTE_P | PTE_W | PTE_U;
}
return &pgtab[PTX(va)];
}

static int mappages(pde_t *pgdir, void *va, uint size, uint pa, int perm) {
char *a, *last;
pte_t *pte;

a = (char*)PGROUNDDOWN((uint)va);
last = (char*)PGROUNDDOWN(((uint)va) + size - 1);
for(;;) {
    if((pte = walkpgdir(pgdir, a, 1)) == 0)
        return -1;
    if(*pte & PTE_P)
        panic("remap");
    *pte = pa | perm | PTE_P;
    if(a == last)
        break;
    a += PGSIZE;
    pa += PGSIZE;
}
return 0;
}

static struct kmap {
void *virt;
uint phys_start;
uint phys_end;
int perm;
} kmap[] = {
{ (void*)KERNBASE, 0,             EXTMEM,    PTE_W}, //                 I/O space
{ (void*)KERNLINK, V2P(KERNLINK), V2P(data), 0},     // kern text+rodata
{ (void*)data,     V2P(data),     PHYSTOP,   PTE_W}, // kern data+memory
{ (void*)DEVSPACE, DEVSPACE,      0,         PTE_W}, // more devices
};

pde_t* setupkvm(void) {
pde_t *pgdir;
struct kmap *k;

if((pgdir = (pde_t*)kalloc()) == 0)
    return 0;
memset(pgdir, 0, PGSIZE);
if (P2V(PHYSTOP) > (void*)DEVSPACE)
    panic("PHYSTOP too high");
for(k = kmap; k < &kmap[NELEM(kmap)]; k++)
    if(mappages(pgdir, k->virt, k->phys_end - k->phys_start,
                (uint)k->phys_start, k->perm) < 0) {
        freevm(pgdir);
        return 0;
    }
return pgdir;
}

void kvmalloc(void) {
kpgdir = setupkvm();
switchkvm();
}

main 調用 kvmalloc，創建並切換到一個擁有內核運行所需的 KERNBASE 以上映射的頁表。這裏的大多數工作都是由 setupkvm 完成的。首先，它會分配一頁內存來放置頁目錄，然後調用 mappages 來建立內核需要的映射，這些映射可以在 kmap 數組中找到。這裏的映射包括內核的指令和數據，PHYSTOP 以下的物理內存，以及 I/O 設備所占的內存。setupkvm 不會建立任何用戶內存的映射，這些映射稍後會建立。

mappages做的工作是在頁表中建立一段虛擬內存到一段物理內存的映射。它是在頁的級別，即一頁一頁地建立映射的。對于每一個待映射虛擬地址，mappages 調用 walkpgdir 來找到該地址對應的 PTE 地址。然後初始化該 PTE 以保存對應物理頁號、許可級別（PTE_W 和/或 PTE_U）以及 PTE_P 位來標記該 PTE 是否是有效的。

walkpgdir 模仿 x86 的分頁硬件為一個虛擬地址尋找 PTE 的過程。walkpgdir 通過虛擬地址的前 10 位來找到在頁目錄中的對應條目，如果該條目不存在，說明要找的頁表頁尚未分配；如果 alloc 參數被設置了，walkpgdir 會分配頁表頁並將其物理地址放到頁目錄中。最後用虛擬地址的接下來 10 位來找到其在頁表中的 PTE 地址。
